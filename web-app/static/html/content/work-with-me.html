My stack as a solo developer

What I use every day — explained without jargon

EHTML — pages that stay simple

I extend ordinary web pages — I don't invent something trendy that expires in two months. With EHTML, a page can show live information, handle forms, update one section without reloading the whole screen, and pull in content from other pages — the everyday things most sites need, written as simple page markup instead of a heavy "modern frontend" stack. No compile step, no fragile toolchain — the browser does the work, so pages stay fast, accessible, and cheap to maintain, even for complex products.

e-ui — consistent, professional UI

Buttons, forms, panels, and dialogs that look and behave the same across the product — professional, consistent, easy to use. I don't reinvent design from scratch every year; I polish what the web already provides so everything feels like one product, not a patchwork of plugins. This page uses it; so do my products.

nodes.js — the server side, under control

The same idea behind the scenes: the server that handles logins, billing, scheduling, and the logic your product depends on. Straightforward and built to last — not buried under enterprise framework weight. It runs efficiently on ordinary servers, grows when traffic grows, and when something breaks at 2 a.m. the path from problem to fix is short.

Plus the boring reliable parts: careful database changes, deploying the app the same way every time, and outside services only when you truly need them — taking payments, sending email, things that are foolish to rebuild from scratch. I keep everything consistent and reliable end to end, and I don't plug in a pile of tools when a simple solution already does the job.

I wrote these tools because I got tired of stacks that optimize for conference talks instead of products still running a decade later. Why I build this way →

How I work

What you get when we collaborate — and a few principles I won't bend on
Zero bloat

Frameworks with zero dependencies, aligned with native platforms without artificial complexity. Fast, cheap shipping. Every technical choice serves the product.

Solo execution

Design, architect, implement, and deploy — all by myself. Full-stack development, end to end. First visible results in two weeks. Production-ready MVPs in months, not quarters or years.

One Focus

One exclusive engagement at a time. Direct communication, zero-fluff discovery, then working software.

If it works, don't rewrite it

When performance and security are fine, we don't rebuild because a new language or library version came out. Upgrades happen when there's a real payoff — not on autopilot. The frameworks I use are built for longevity and don't have breaking changes over the years.

No microservices. No SaaS sprawl. No cloud

One coherent system beats twenty services that multiply cost and ops. No managed cloud when a VPS does the job — same uptime, a fraction of the bill. And if something takes thirty lines of code, we don't add another monthly subscription.

Test what users actually do

End-to-end tests on real product flows — not mock-heavy unit tests that pass while production breaks. We verify features work, not that fake data behaved.

Real problems only

We don't invent complexity because it's fashionable — replacing a straightforward solution with layers of abstraction, hooks, and ceremony that slow delivery and burn budget.

Ship fast, change fast

I prefer stacks where iteration stays quick as the codebase grows — no hour-long builds between a fix and seeing it live. Speed of change is a business feature.

Explicit over clever

Code should say what the program does — direct, readable, no magic. Easier to maintain, easier to trust, cheaper to hand off or extend later.
