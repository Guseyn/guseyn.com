My stack as a solo developer

What I actually use — in plain language

EHTML — HTML that does more

I extend HTML. I didn't invent a new file format that will be dead in two years. EHTML extends what pages already have and adds more: repeat bits of markup for lists, handle complex forms without wiring up JavaScript, update part of a page without a full reload, pull in shared HTML from elsewhere. Most of what you'd normally solve with a pile of JavaScript — here it's just markup. No build step. No bundling, transpiling, or any of that shit.

e-ui — design system on top of the web

Buttons, forms, panels, dialogs — same look and behaviour everywhere, so the product doesn't feel patched together. I didn't invent a new design language. I just extended what browsers already give you. I build with it because pages come out looking designed without reinventing CSS from scratch.

nodes.js — the server side

Same idea on the server. Logins, billing, scheduling, APIs — Node.js, without burying it under ten frameworks. It has a built-in Cluster API: when traffic grows, the app uses all the cores on the machine instead of one; if something crashes, it comes back on its own; you can restart for a deploy without taking the whole site offline. It runs on an ordinary machine, you can scale it when you need to, and when something breaks at 2 a.m. you're not grep-ing through six abstraction layers to find the bug.

And the boring parts: database migrations done carefully. Same deploy process every time. Stripe for payments, Postmark for email — third-party stuff only where rebuilding it would be stupid. I don't wire in fifteen SaaS tools when thirty lines of code does the job.

I wrote EHTML and e-ui because I got tired of stacks that look great in a conference talk and fall apart when you try to change them six months later. Why I build this way →

How I work

How I actually work — a few things I won't bend on
Zero bloat

No dependencies for the sake of dependencies. If the browser or Node already does it, I use that. Less stuff to break, less stuff to pay for.

Solo execution

I design it, I write it, I deploy it. No hand-offs between frontend guy, backend guy, and DevOps guy — because that's me. You can usually see something working in about two weeks.

One Focus

One client at a time. When I'm on your project, I'm not splitting attention across three Slack workspaces. We talk directly, figure out scope, then I write code.

If it works, don't rewrite it

If it works and it's fast enough, we don't rewrite it because a new framework version dropped. Upgrades happen when there's a real reason — not because a blog post told us to.

No microservices. No SaaS sprawl. No cloud

One system, not twenty microservices. No managed cloud when a VPS does the job. And if something takes thirty lines of code, we don't buy a SaaS subscription for it.

Test what users actually do

End-to-end tests on real flows — login, pay, book a lesson, whatever your users actually do. Not unit tests on mocked data that pass while the site is broken.

Real problems only

We don't add layers because everyone else is adding layers. If a simple solution works, we keep it simple. Fashion passes; your codebase stays.

Ship fast, change fast

No hour-long builds between a fix and seeing it live — deploys take a couple of seconds, not hours. If changing the product is slow, the product stops improving — that's a business problem, not just a dev problem.

Explicit over clever

Code should read like code. No magic, no clever one-liners that need a comment to explain. You or someone else should be able to open a file and know what it does.
